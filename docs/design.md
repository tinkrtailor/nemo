## Problem Statement

Spec-driven agentic development workflow (PRD -> spec -> hardening -> implementation -> review) works but is bottlenecked by local machine resources. An M2/32GB maxes out at 6 parallel agents. Team members with weaker machines hit lower ceilings. The iteration loops (3-4 rounds of spec hardening, 3-4 rounds of implementation review) are manual and sequential. The team needs shared infrastructure where agent execution is decoupled from developer machines.

## What Makes This Cool

A convergent, multi-model adversarial build system for software changes. You push a spec, the system runs implement -> review cycles until the reviewer finds nothing wrong, and you get a clean PR. Each engineer on the team gets N parallel execution slots on shared infrastructure. The unique primitive: cross-model adversarial review (Claude implements, OpenAI reviews) produces higher quality than single-model approaches. Nobody else is building this loop as the core abstraction.

Codex called it: "GitHub Actions meets Bazel meets evals, but for spec -> patch -> review -> merge." That's the product.

## Constraints

- Must own the infrastructure (no Coder, Codespaces, managed platforms)
- Per-monorepo clustering (one cluster serves one monorepo)
- Per-engineer git identity (Alice's agents commit as Alice)
- Convergent loops (exit on clean verdict, not fixed iteration count)
- Must work for a team of 3-5 engineers initially
- Budget-conscious: Hetzner VPS, not GKE/EKS
- Claude Max + OpenAI Pro subscriptions as model backends
- Decoupled from any specific project (works for any monorepo)

## Premises

1. The real bottleneck is workflow automation + compute, not just compute. Moving to remote infra solves parallelism, but the convergent loop is what eliminates manual iteration.
2. The product is the orchestration layer (spec -> convergent loop -> PR), not the infrastructure. k3s on Hetzner is the substrate.
3. Per-monorepo clustering is the right abstraction. Each monorepo gets its own cluster with its own agent configurations, model preferences, and review policies.
4. Self-improving agents (review feedback -> better implementations) is V2+. V1 proves the convergent loop.
5. Multi-model adversarial review is the key differentiator. Single-model review has blind spots. Cross-model review catches more.

## Cross-Model Perspective

Codex cold read identified the core reframe: the product is not "K8s for agents" but a "repo-native adversarial build system." The infrastructure is commodity; the orchestration and quality scoring are the product.

Codex's weekend V1: one repo, one branch, one workflow (implement -> review-A -> review-B). K8s Jobs on k3s, SQLite control plane, hardcoded two-model roles. If it can run 10 parallel loops for one monorepo without human babysitting, you have the wedge.

### Future Vision (V2+)

Spec compiles into a DAG. Multiple implementers race on the same spec. Multiple reviewers attack independently. A judge selects the best patch based on test results, review scores, and token cost. The system learns which model-role pairings work best per monorepo. None of this is V1.

## Approaches Considered

### Approach A: k3s Control Plane (selected foundation)
k3s on Hetzner VPS. SQLite/Postgres control plane. Agent loops as K8s Jobs. Each job: clone repo at SHA, run in standard image, output patch + logs + test results. Minimal web dashboard. Hardcode implementor (Claude) + reviewer (OpenAI). Terraform module to stamp out new clusters.
- Effort: S/M. Risk: Low.
- Pro: Own the infra, simple, cheap, fast to ship.
- Con: No DAG abstraction, hardcoded stages.

### Approach B: Spec DAG Engine
DAG compiler that turns a spec into stages. Each stage dispatches to the right model. Adds stage routing, artifact lineage, reviewer verdict scoring.
- Effort: M. Risk: Medium.
- Pro: The "real product" from day one.
- Con: More upfront design before proving the loop works.

### Approach C: Git-native CLI (deferred)
CLI tool using git worktrees locally or over SSH. No K8s dependency.
- Effort: S. Risk: Low.
- Con: Doesn't solve parallelism ceiling.

## Recommended Approach

**Nemo** (hybrid A+B): Build the k3s infrastructure with the convergent loop as the core primitive, but design the job dispatch to be stage-aware from the start. Not a full DAG engine, but structured enough that stages (implement, review, test) are first-class concepts.

### Core Components

**1. Control Plane (Rust binary, runs as k3s deployment)**
- REST API for job submission, status, logs
- Loop engine: dispatches implement job, waits for completion, dispatches review job, evaluates verdict, loops or exits
- Job state machine: PENDING -> HARDENING -> IMPLEMENTING -> TESTING -> REVIEWING -> CONVERGED | HARDENED | SHIPPED | FAILED
- Full pipeline: `nemo start --harden spec.md` runs harden loop first, then implementation loop
- Skip hardening: `nemo start spec.md` skips to implementation (for pre-hardened specs)
- Harden only: `nemo harden spec.md` runs spec hardening loop, merges hardened spec PR, stops
- Auto-merge: `nemo ship spec.md` runs implementation loop and auto-merges on convergence
- Per-engineer credential store (git SSH keys / PATs + model subscription credentials as K8s secrets)
- Postgres for state (eng review decided Postgres over SQLite for concurrent access from split control plane). Postgres runs as k3s pod with PVC.

**2. Agent Job Image (container image with claude-code + opencode pre-installed)**
- claude-code: Anthropic's CLI agent for Claude models (implementer). Headless via `claude -p --output-format stream-json`
- opencode: OpenCode CLI (opencode.ai, github.com/anomalyco/opencode) (reviewer). Headless via `opencode run --format json` or persistent `opencode serve` with REST API
- Clones repo from shared bare repo (see Resource Model) into a worktree
- Receives: spec file path, stage (implement/review/test), model config, git identity, feedback file (if review round > 1)
- Outputs: branch tip SHA (not a patch), affected services manifest, test results, review verdict (JSON), logs, token usage
- The implement job outputs an `affected_services` list (parsed from changed file paths). The test job uses this to run only the relevant test suites.
- The review job diffs against the base branch (`git diff base...sha`) to evaluate the full change
- Git commits attributed to the engineer who submitted the job
- Multi-round feedback via session continuation: `claude -p --resume $SESSION_ID` (implementer) and `opencode run -s $SESSION_ID` (reviewer). Each round is a clean process invocation (clean k8s Job), session state persisted on PVC. Alternative: `opencode serve` runs as a persistent sidecar, control plane talks to it via REST API (`POST /session/:id/prompt_async`), avoiding process spawn per round.
- Review verdict enforced via OpenCode's SDK-level JSON Schema structured output (retries on validation failure)
- **Image strategy:** Nemo ships a slim base image (git, claude-code, opencode, common runtimes, `ghcr.io/anomalyco/opencode` as base for reviewer). Each monorepo provides a `Dockerfile.nemo` (or `nemo.image` in `nemo.toml`) that extends the base with project-specific toolchains (Rust, JVM, Node, Solidity/Foundry, Python, etc.). `nemo init` generates a starter Dockerfile based on detected toolchains. This keeps Nemo stack-agnostic.

**3. nemo CLI — three verbs**

| Command | What it does | Terminal state |
|---------|-------------|----------------|
| `nemo harden spec.md` | Harden spec, merge spec PR | HARDENED |
| `nemo start spec.md` | Implement, create PR | CONVERGED |
| `nemo start --harden spec.md` | Harden first, approval gate, then implement | CONVERGED |
| `nemo ship spec.md` | Implement + auto-merge | SHIPPED |
| `nemo ship --harden spec.md` | Harden + implement + auto-merge | SHIPPED |

`--harden` is the only shared flag. It prepends the hardening phase to `start` or `ship`. On `ship`, it skips the approval gate.

Other commands:
- `nemo status` -- show all running loops for current user
- `nemo status --team` -- show all loops across the team
- `nemo logs <loop-id>` -- stream logs for a specific loop
- `nemo inspect <branch>` -- view any teammate's in-progress work (branch passed as query param to API)
- `nemo cancel <loop-id>` -- cancel a running loop
- `nemo init` -- scan monorepo, detect services, generate starter `nemo.toml`. Engineer reviews and commits.
- `nemo auth` -- pipe local model credentials (Claude Max session, OpenAI Pro tokens) to the control plane. Stored as K8s secrets scoped to the engineer. The engineer extends their own subscription into the cluster, same model as local worktrees.
- `nemo config` -- configure personal model preferences, review thresholds (writes `~/.nemo/config.toml`)
- Auth: API key to control plane (mTLS deferred to V2)

### Two Convergent Loops

The system has two distinct convergent loops, run in sequence:

1. **Spec Hardening Loop**: Draft spec → adversarial audit → revise → audit → until clean. This is the planning phase. The auditor and the author are different models (cross-model adversarial, same principle as implementation review). Output: a hardened spec ready for implementation.

2. **Implementation Loop**: Implement → test → review → fix → review → until clean. This is the execution phase. Output: a PR with passing tests and clean review.

Both loops use the same convergent primitive (do work → adversarial check → loop or exit). The control plane treats them as stages in a pipeline: `HARDEN → IMPLEMENT`. An engineer can submit a raw spec and get a hardened spec + PR, or submit a pre-hardened spec and skip to implementation.

### Spec Hardening Loop Logic

```
fn run_harden_loop(spec_draft, engineer):
    branch = create_branch("spec/" + spec_draft.name, engineer)
    spec_file = copy_spec_to_branch(spec_draft, branch)
    round = 0

    loop:
        round += 1

        // Audit stage: adversarial review of the spec
        audit_job = dispatch_job(
            stage: SPEC_AUDIT,
            spec: spec_file,
            branch: branch,
            model: engineer.preferred_reviewer || "openai",
            credentials: engineer.model_credentials,
            context: "Audit this spec for ambiguity, missing edge cases, "
                     + "unresolved dependencies, testability, and feasibility.",
        )
        wait(audit_job)

        audit_verdict = audit_job.output.verdict
        if audit_verdict.clean:
            return (HARDENED, spec_file)

        if round >= MAX_HARDEN_ROUNDS (default: 10, configurable):
            return (NEEDS_HUMAN_REVIEW, spec_file, audit_verdict.issues)

        // Revise stage: fix spec based on audit findings
        revise_job = dispatch_job(
            stage: SPEC_REVISE,
            spec: spec_file,
            branch: branch,
            model: engineer.preferred_implementor || "claude",
            credentials: engineer.model_credentials,
            feedback: audit_verdict.issues,
        )
        wait(revise_job)
        spec_file = revise_job.output.updated_spec
```

### Implementation Loop Logic

```
fn run_loop(spec, engineer):
    branch = create_branch(spec.name, engineer)
    sha = get_head_sha(branch)
    round = 0

    loop:
        round += 1

        // Implementation stage
        impl_job = dispatch_job(
            stage: IMPLEMENT,
            spec: spec,
            branch: branch,
            sha: sha,
            model: engineer.preferred_implementor || "claude",
            git_identity: engineer.git_config,
            credentials: engineer.model_credentials,
            feedback: feedback_file || null,  // null on first round
        )
        wait(impl_job)
        sha = impl_job.output.new_sha

        // Test stage (run only affected service test suites)
        test_job = dispatch_job(
            stage: TEST,
            branch: branch,
            sha: sha,
            affected_services: impl_job.output.affected_services,
        )
        wait(test_job)
        if test_job.failed:
            // feed test failures back to next implement round
            continue

        // Review stage
        review_job = dispatch_job(
            stage: REVIEW,
            spec: spec,
            branch: branch,
            sha: sha,
            model: engineer.preferred_reviewer || "openai",
            context: impl_job.output.patch,
        )
        wait(review_job)

        verdict = review_job.output.verdict
        if verdict.clean:
            create_pr(branch, engineer, round_count: round)
            return CONVERGED

        // MAX_ROUNDS is a safety valve, not an expected exit condition.
        // Complex specs may legitimately need 10+ rounds. The default is
        // generous (15) and configurable per-spec or per-engineer.
        // The loop SHOULD exit via convergence (clean verdict), not timeout.
        if round >= MAX_ROUNDS (default: 15, configurable):
            create_pr(branch, engineer,
                      status: NEEDS_HUMAN_REVIEW,
                      remaining_issues: verdict.issues)
            return MAX_ROUNDS_EXCEEDED

        // Feed review issues back to next implementation round.
        // The review verdict's `issues` array is written to a JSON file
        // at `.agent/review-feedback-round-{N}.json` in the worktree.
        // The next implement job receives the spec file path AND the
        // feedback file path. The agent prompt includes:
        // "Fix the following issues found in review: {issues}"
        // This is prompt injection, not spec mutation.
        feedback_file = write_feedback(verdict.issues, round)
        spec_context = spec + feedback_file
```

### Resource Model

Target: 15 concurrent agent jobs (3 engineers x 5 loops). Each agent job runs a CLI tool (claude-code or codex) that streams from an external API, so the job itself is not GPU-bound. Primary resource consumption is CPU (for git operations, test execution) and RAM (CLI runtime, repo checkout).

| Job type | RAM estimate | CPU | Notes |
|----------|-------------|-----|-------|
| Implement (claude-code/codex) | ~2 GB | 0.5 vCPU (API-bound) | Mostly waiting on model API |
| Review (claude-code/codex) | ~2 GB | 0.5 vCPU (API-bound) | Reads diff, outputs verdict |
| Test (lightweight: npm, cargo, phpunit) | ~2-3 GB | 1 vCPU | Standard test runners |
| Test (heavyweight: sbt/JVM) | ~4-6 GB | 2 vCPU | JVM alone consumes 2-4 GB |
| Disk per worktree | ~2 GB | - | Shared bare repo reduces clone overhead |

Worst case: 15 concurrent jobs, 5 of which are JVM test jobs = ~50 GB RAM. The control plane scheduler should enforce a **concurrent test job limit** (default: 3 JVM jobs) to prevent memory exhaustion. Implement and review jobs are lightweight and can run at higher concurrency.

Hetzner sizing: Single **CCX43** (16 vCPU, 64 GB RAM, 360 GB disk, ~EUR 72/mo) handles 15+ concurrent jobs on one box. k3s supports multi-node (server + agent topology), so when you outgrow one machine, add agent nodes with `k3s agent --server https://...`. The control plane code doesn't change. Start single-node, design for multi-node. The terraform module provisions one box with a `node_count` variable for future expansion.

The cluster maintains a **shared bare repo** (PersistentVolume) with a cron job running `git fetch` every 60s. Agent jobs create worktrees from this bare repo instead of cloning, reducing per-job startup from minutes to seconds. Note: `git worktree add` takes a lock on the bare repo. With 15 concurrent jobs, worktree creation is effectively serialized. At <1s per operation, worst case is ~15s queue time, which is acceptable. The control plane serializes worktree create/delete operations through a simple mutex.

### Failure Handling

| Failure | Detection | Response |
|---------|-----------|----------|
| Job OOM / pod eviction | K8s job status | Retry up to 2 times with backoff (30s, 120s). On 3rd failure, mark loop FAILED, notify engineer. |
| Model API timeout | Job exits non-zero after 10 min timeout | Retry once. If second attempt times out, mark loop FAILED. |
| Job stuck (no output for 15 min) | Control plane watchdog | Kill job, retry once. |
| Partial commits on failure | Job cleanup hook | On non-zero exit, `git reset --hard` to the SHA the job started with. Worktree is ephemeral, so no state leaks. |
| Control plane crash | K8s deployment restarts | Loop state is in Postgres. On restart, resume all IMPLEMENTING/REVIEWING loops from last known state. Jobs are nondeterministic (model-driven), but start from pinned SHA with same inputs. |

### Review Verdict Schema

The review job outputs a JSON file at `.agent/review-verdict.json`:

```json
{
  "clean": false,
  "confidence": 0.85,
  "issues": [
    {
      "severity": "high",
      "file": "backend/src/Invoice.php",
      "line": 42,
      "description": "Missing null check on customer_id before database lookup",
      "suggestion": "Add early return with 400 response if customer_id is null"
    }
  ],
  "summary": "Implementation covers the happy path but misses two edge cases in error handling.",
  "token_usage": { "input": 45000, "output": 3200 }
}
```

`clean: true` means zero issues. The loop engine checks `verdict.clean` to decide whether to continue or converge. The `confidence` field is informational for V1 (used by the judge in V2+ for multi-reviewer scoring).

### Branch Isolation and Concurrency

- Each loop gets its own branch: `agent/{engineer}/{spec-slug}` (e.g., `agent/alice/invoice-cancel`).
- Only one loop may be active per branch. Submitting a spec that maps to an existing active branch is rejected.
- Agent jobs operate on ephemeral worktrees created from the shared bare repo. Worktrees are deleted after each job completes.
- If an engineer pushes manually to a branch that has an active loop, the control plane detects the SHA mismatch on the next job dispatch. The loop pauses and notifies the engineer: "Branch diverged. Resume or cancel?"
- Two loops for the same engineer touching different branches cannot conflict (separate worktrees, separate branches).

## Open Questions

1. **Model credentials: solved.** Claude Code runs headless via `claude --print --output-format stream-json` with Max subscription auth. The `~/.claude/` session directory is mounted from a K8s Secret directly into the agent container's HOME (`/work/home/.claude/`). Claude Code reads its own session auth and connects directly to `api.anthropic.com`. The sidecar model proxy (`:9090`) is used only for OpenAI (injects `Authorization: Bearer` header for opencode). `nemo auth` copies credentials into K8s secrets scoped to the engineer. **Fallback:** API keys (Anthropic API + OpenAI API) for teams that prefer direct API access over subscription piping.
2. **Cost tracking**: Token usage per loop, per engineer, per day. The review verdict schema includes `token_usage`. The control plane should aggregate this. Where does cost data surface? Dashboard? CLI? Daily email?
3. **Spec format**: Do existing specs in `specs/` work as-is, or does the system need a wrapper format with metadata (target services, test commands, review thresholds, max rounds)?
4. **Network security: solved.** Auth sidecar architecture. Agent containers get open internet (for docs, deps, research) but NO mounted secrets. Model API auth and git push proxy through a sidecar container (localhost:9090, :9091). Sidecar also runs a transparent egress logger (:9092) that logs all outbound traffic from the agent container for audit. Secrets never touch the agent container's filesystem. Malicious code can reach the internet but has nothing to exfiltrate.
5. **Naming**: Resolved. **Nemo**. Latin for "nobody." Three verbs: `nemo harden`, `nemo start`, `nemo ship`.

## Success Criteria

- 3 engineers running 5 parallel convergent loops each on shared k3s cluster
- Average convergence in 2-4 rounds (implement -> review cycles)
- Zero manual intervention for loops that converge
- PR created automatically on convergence, attributed to the submitting engineer
- Terraform module provisions a new cluster in under 10 minutes
- Total infrastructure cost under $100/month on Hetzner (excluding model API costs)
